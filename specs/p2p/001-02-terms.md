# 001-02 — 术语表（Terms & Glossary）

> 状态：**v0.1 — 初版**。本文是 P2P 系列文档（001 ~ 006）的术语索引与中英文 / 代码命名对照表。所有跨文档术语争议、命名歧义、以及"我们自己造词"与"借用既有名词"的边界都在这里裁定。
>
> 维护规则：
> - 任何在 P2P 文档中**首次引入**或**重命名**的术语，必须同步在此登记。
> - ==**带 ★ 标记**==：BitRouter 协议**自有**术语（不存在于 iroh/MPP/W3C/CAIP 等上游规范）。
> - ⊕ 标记：借用上游、但**含义经我们扩展**的术语（含义偏离上游时尤其要看 §"备注"）。
> - 其他：直接借用、含义与上游一致。
>
> 与本文配套的总入口仍是 [`001-01-overview.md`](./001-01-overview.md)。

## 0. 命名约定（Conventions）

| 维度 | 约定 |
|---|---|
| 文档正文中文称谓 | 优先使用中文术语（如"提供方"），但**首次出现**或**关键引用**处必须括注英文 / 代码名（如"提供方（Provider，`provider_id`）"） |
| 英文术语 | PascalCase / 单数（如 `Provider`、`PaymentGateway`） |
| 代码字段 / JSON key | snake_case（如 `provider_id`、`endpoint_id`、`accepted_pgws`） |
| Rust 类型 | PascalCase + 上游约定（如 `EndpointId`、`PaymentRequirements`） |
| 链上地址 | CAIP-10 全称（如 `eip155:4217:0x...`、`solana:.../USDC`） |
| 哈希算法 | 小写连字符（如 `sha256-canonical-json`） |
| ALPN / 协议字符串 | 小写、斜杠分段（如 `bitrouter/p2p/0`） |

**避免引入的命名**（历史包袱，不要再使用）：

| 弃用名 | 替代 | 弃用原因 |
|---|---|---|
| `logical_provider_id` | `provider_id` | 引入"逻辑/物理"二分增加歧义；root pubkey 本身已是逻辑身份 |
| `logical_pgw_id` | `pgw_id` | 同上 |
| `logical_id` | `provider_id` / `pgw_id` | 同上 |
| `provider_endpoint_id`（指身份层时） | `provider_id` | 把 "provider 身份" 与 "iroh endpoint" 混淆 |
| `pgw_endpoint_id`（指身份层时） | `pgw_id` | 同上 |
| `accepted_payment_gateways` | `accepted_pgws` | 字段过长；保留旧名仅在 005 v0.7 历史段落出现 |
| `max_receivable_to_cloud` | `max_receivable_to_pgw[pgw_id]` | "Cloud" 是默认实例，不应硬编码进字段名 |
| Cloud / BitRouter Cloud（作为协议角色） | PGW（角色） + "BitRouter Cloud"（作为 PGW 默认实例的产品名） | 协议层去 Cloud 化 |
| 自定义事件流 / `PUBLISH` / `REVOKE` / `ROTATE` 事件 | snapshot 替换 + `seq` 递增（详见 003 §8.1） | 显式排除事件流架构 |

## 2. 身份与密钥（Identity & Keys）

==**核心模型 = 两层身份**==（详见 [`003 §2.1`](./003-l3-design.md)）。

| 中文 | 英文 | 代码 | 标记 | 含义 |
|---|---|---|---|---|
| 提供方根身份 | Provider Identity | `provider_id` | ★ | ed25519 **root pubkey**，Provider 的长期身份；签整份 Registry snapshot；冷密钥，离线托管 |
| 网关根身份 | PGW Identity | `pgw_id` | ★ | ed25519 root pubkey，PGW 的长期身份；同上语义 |
| 实例端点 ID | Endpoint ID | `endpoint_id` | ⊕ | iroh `EndpointId`（也是 ed25519 pubkey）；per-process 热密钥；用于 QUIC 拨号 |
| iroh 端点 | iroh Endpoint | iroh `Endpoint`、`EndpointId` | （iroh 原词） | L1/L2 层原语；本文档 L1/L2 章节直接保留英文 `EndpointID` 写法 |
| 链上地址 | Chain Address | `chain_addr` | ⊕ | 由 `provider_id`/`pgw_id` HD 派生（BIP32/SLIP-10）的链上收款 / 签名地址；CAIP-10 编码 |
| 支付密钥 | Payment Key | `payment_key` | ★ | 上述 chain_addr 的私钥；待 [`006-l3-auth`](./006-l3-auth.md) 规范化 |
| Tempo 支付 EOA | Tempo Payment EOA | `tempo_eoa` | ⊕ | ==Consumer 在 Tempo session intent 下需要持有的 secp256k1 EOA==；用于按 EIP-712 签 `TempoStreamChannel` voucher（详见 [`004-02 §4`](./004-02-payment-protocol.md) / R9）；DID 形式 `did:pkh:eip155:4217:0x...`；与 BitRouter ed25519 身份**完全解耦**（ed25519 仍负责 node identity / snapshot / `Payment-Receipt` / order extension 签名） |
| 密钥轮换 | Root Key Rotation | snapshot 内 `next_root_pubkey` | ★ | snapshot 嵌入下一把 root；过渡 snapshot 由旧+新 root 联签（详见 003 §8.2） |
| Capability Grant | Capability Grant | `capability_grant_signature` | ★ | 由 `pgw_id` root 签发的 PGW endpoint 授权凭证（详见 004-01 §3.1） |
| 雇主签名 / 委托 | （旧名） | — | — | 在两层身份下不再单独立名——root 签 snapshot 自动覆盖全部 endpoint |

> ==**两层身份退化**==：002 prototype MVP 是 `provider_id == endpoint_id` 的退化形态。这是**协议合法状态**，不是历史遗留。

## 8.5 身份字符串编码（Identity String Encoding）

==**统一编码规则**==（R5）：所有 BitRouter 协议消息、URL 路径、CLI 参数、snapshot、结构化日志中的公钥字符串只有一种线上形式：

```text
<algo>:<z-base32-lower-no-pad(pubkey_bytes)>
```

z-base32 字母表锁定为 Zooko 变体：

```text
ybndrfg8ejkmcpqxot1uwisza345h769
```

它不是 RFC 4648 base32（`a-z2-7`）。实现必须按 5-bit 分组使用上表编码，禁止 padding，禁止大小写折叠。

| algo tag | 公钥字节 | z-base32 字符数 | 状态 |
|---|---:|---:|---|
| `ed25519` | 32 | 52 | v0/v1 唯一启用；Provider / PGW root key、iroh endpoint key |
| `secp256k1` | 33（compressed） | 53 | 预留 |
| `secp256r1` | 33（compressed） | 53 | 预留 |
| `sr25519` | 32 | 52 | 预留 |
| `bls12_381_g1` | 48 | 77 | 预留 |

未列入本表的 `<algo>` 字符串均为非法；新增签名算法必须先修订本表。

| 场景 | 字符串形式 | 备注 |
|---|---|---|
| BitRouter node identity（`provider_id` / `pgw_id` / `endpoint_id`） | `<algo>:<z-base32-lower-no-pad>` | v0/v1 只允许 `ed25519:<52 chars>` |
| Tempo / EVM 链上 EOA 身份（Consumer 支付账户、Tempo voucher 签名者） | `did:pkh:eip155:<chainId>:0x<40 hex>` | 标准 DID PKH；Consumer 在 Tempo session intent 下需要持有一把 secp256k1 EOA（Tempo 钱包），与其 BitRouter ed25519 身份解耦 |
| iroh API 边界 | iroh 内部类型 ↔ `ed25519:<z-base32>` | iroh `PublicKey::Display` 的 hex 只允许在 SDK 内部边界出现；进入 BitRouter 协议字段、CLI、日志、snapshot 前必须转为 `ed25519:<z-base32>` |

==**iroh hex 边界**==：`endpoint_id` 是 ed25519 公钥的特例。调用 iroh API 时若拿到 hex（64 字符），必须在 SDK 边界处用 `from_z32` / `to_z32` 等价逻辑立即转码为 `ed25519:<z-base32>`；hex 表示禁止外泄到协议消息、CLI、日志、snapshot。向 iroh 传 `EndpointId` 时在同一边界反向转码。

弃用形式：

| 弃用 | 替代 | 说明 |
|---|---|---|
| `hex(endpoint_pubkey)` / 裸 hex 公钥 | `<algo>:<z-base32-lower-no-pad>` | iroh `Display` hex 不得作为协议字符串 |
| RFC 4648 base32（`a-z2-7`） | Zooko z-base32 | 字母表不同，互不兼容 |
| 裸 base58 / base64 / base64url / multibase | `<algo>:<z-base32-lower-no-pad>` | 协议层不使用 multibase 前缀 |
| `did:key:z...` | `<algo>:<z-base32-lower-no-pad>` | 不引入 W3C DID / multicodec parser 作为节点身份依赖 |
| 裸 `0x...` hex（无 DID 前缀）作 EOA 字段值 | `did:pkh:eip155:<chainId>:0x...` | EVM 支付账户用 DID PKH，不与 BitRouter node identity 混用 |
